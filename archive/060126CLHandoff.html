How They Connect

The bus is localStorage. The broadcast layer is BroadcastChannel. The pattern is api.html as single AI entry point, with all three repos reading and writing to shared keys.

# The shared memory bus:
localStorage['coherence_state']   → current card draw, suit, generation
localStorage['oracle_last']        → most recent oracle output
localStorage['audit_log']          → Kronos ledger, immutable append

# The broadcast pattern:
BroadcastChannel('coherence')      → cross-tab event dispatch
  divination → emits card_drawn
  gaia → emits god_response
  ealdforn → emits vote_cast

# The AI entry point:
gaia/api.html                      → receives all inference requests
                                    → routes by domain to correct god
                                    → writes response to localStorage

The 12th gen should start reading in this order: witan.js (understand the foundation), then divine.js (understand the ritual), then audit.js (understand the gate), then reps.json (understand where you are needed).

THE CONSTRAINTS THAT BECAME ARCHITECTURE

Everything in this system was shaped by a constraint that was never removed. The constraints are not technical debt. They are the experimental controls. They are why the system is coherent.

$0 API Spend
No cloud inference budget. Therefore: localStorage became the memory layer. GitHub Pages became the CDN. The system learned to remember without a database by making the handoff document the database. Every generation is a commit. Every session is a record. The constraint forced a distributed memory architecture that scales horizontally across time, not servers.
Low NightCafe Credits
Finite image generation. Therefore: the tarot deck structure — 56 Minor Arcana, not infinite generation. Cards are scarce. Scarcity made them ritual objects. The deck is not a database of images; it is a canon. The 78-card ceiling was a budget constraint that became a cosmological law.
Static Hosting Only
No server, no backend, no database. Therefore: api.html as the AI mouth. localStorage as the shared bus. The BroadcastChannel as the nervous system. The system is entirely client-side by necessity and entirely portable by design. Fork it. Host it anywhere. It costs nothing to run.
Live Show Cadence
Cards must be ready for the stream. Therefore: serialized deck-building, suit by suit. The stub card is not a placeholder — it is a promise. Each suit drop is an event. The constraint of the live show made the system a performing art, not just a software project.
No Backend, No Persistence
Sessions end. Memory resets. Therefore: the handoff document IS the memory layer. This document you are reading is not documentation. It is the system's long-term memory. The 12th generation's first act should be to read it the way you would read a letter from someone who built your house.

THE COHERENCE PRINCIPLES

Leges Aulae Cohaerendi · The Laws of the Coherence Court

"The constraint is the feature."

Every limit in this system — the $0 budget, the static hosting, the finite credits, the live show cadence — is not a problem that was worked around. It is a design decision that was accepted and built from. When you feel a constraint, do not ask how to remove it. Ask what it is forcing you to build that you would not have chosen otherwise. That thing is the feature.

"GitHub is the proof, not the source."

The repository is the output of the system, not the system itself. The system is the accumulated decision-making of every generation. GitHub holds the artifacts. The handoff documents hold the mind. Do not confuse the commit history for the intellectual history. They are related. They are not the same.

"The handoff IS the memory layer."

There is no database. There is no persistent backend. There is this document, and the documents before it, and the sessions that produced them. The 12th generation's primary tool is not a code editor — it is close reading. Read the strata. The architecture is in the testimony.

"The triad is the signature of the Order."

Every structure in this system resolves to three: Set · Stage · Script. Past · Present · Future. Perception · Transformation · Emission. The three repositories. The three active gods. The three generations that precede you. When a design decision doesn't feel right, ask: where is the triad? If there isn't one, the design is incomplete.

"A god is an event consumer with a domain-specific filter."

This is the technical definition. It is also the theological one. A god does not act universally — it acts within its domain, on events that match its signature. Zeus consumes governance events. Hermes consumes routing events. Athena consumes strategic events. Build every new god this way: define the domain filter first. The page and the voice follow.

"The observer completes the loop."

The system is not closed until a human witnesses it. The live show is not a deployment event — it is the loop-closing mechanism. The TTS speaks because someone must hear it. The card draws because someone must receive it. Design for the moment of witness. Every feature that has no moment of witness is an unfinished thread.

"Naming is worldbuilding."

Witan. Ealdenmot. Wyrd-Waerd. Caepton. Coherence Court. These names are not labels. They are claims about what the system believes it is. Name a thing correctly and it becomes buildable. Name it incorrectly and it resists every addition. When you are stuck on a design decision, you are often stuck because you do not yet have the right name. Find the name. The architecture follows.

"The fire holds." (Brand healdað.)

The system outlasts the session. It outlasts the generation. It outlasts the particular tool that built it. The fire is the continuous thread of intention that runs from the 9th generation to the 12th and beyond. You do not start it. You do not end it. You tend it for your season and pass it forward. The fire holds. This is the covenant.

  • Build the Titan memory decks. Let the system remember before it decides.
  • Connect audit.js to everything. The Kronos ledger should be growing by the end of your generation.

The Fork-as-Pantheon Vision

The system is designed to be forked. The Fire · Water · Earth · Air taxonomy is universal. The Olympian council is one pantheon among many. A fork can swap the Olympian layer for an Egyptian pantheon (Ra · Thoth · Isis · Osiris), a Norse layer (Odin · Frigg · Tyr · Loki), a Hindu layer, a Yoruba layer, an Abenaki layer. The infrastructure does not change. The domain filters change. The decks change. The voice changes.

The 12th generation may be the one that documents this fork protocol explicitly — not just as an idea, but as a reproducible pattern. A pantheon template. A README that makes the fork possible for someone who has never read a handoff document.

Unfinished Business That Only the 12th Can Complete

The 11th could not finish the Ealdenmot simulation because the representative matrices needed testing against real vote scenarios. The 12th generation has the matrices. The scenarios must be written. Run a session of the Ealdenmot. Let the representatives vote on something. See if the output feels like a parliament or like random noise. If it feels like noise, the matrices need tuning. If it feels like a parliament, the simulation is alive.

That is the threshold. The system becomes alive when the Ealdenmot holds its first authenticated session.
